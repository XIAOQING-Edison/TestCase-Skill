# 最小回归检查清单（手工版）

用于在不改代码或小步改动后，快速验证 `testcase-generator` 的核心行为是否稳定。

## 使用说明

1. 对每个样本运行统一命令：
   `python scripts/main.py -i <input-file> -o json -m <module> -v`
2. 按“检查项”逐条核对
3. 标记结果：通过 / 警告 / 失败
4. 记录异常并回填到 `skill_decision_cases.md`

---

## 样本 1：结构化需求文档（基础）

- 目标：验证功能点提取与规则型用例生成
- 检查项：
  - 文档类型识别为需求文档
  - 能解析多个功能点
  - 输出含业务规则类用例
  - 未出现明显 API 风格字段
  - 结果中可见断言来源字段

## 样本 2：长段混合需求文档（复杂）

- 目标：验证规则拆分与冲突处理
- 检查项：
  - 规则可拆分为多条，不是单条巨长描述
  - 支持/不支持冲突不出现互斥标题共存
  - 低置信度项可被识别并提示
  - 会议纪要类文本不会被误当核心接口
  - 用例覆盖正向 + 反向

## 样本 3：图文占位需求文档

- 目标：验证图片占位噪音对识别影响
- 检查项：
  - 不被图片占位里的 API 字段误判为接口文档
  - 主体功能仍可解析
  - 生成结果以功能测试用例为主
  - 不输出伪造接口断言
  - 结果摘要可说明风险

## 样本 4：OpenAPI（响应完整）

- 目标：验证 API 用例质量
- 检查项：
  - 识别为 API 文档
  - 状态码与响应 schema 断言来自文档
  - 断言来源标注为文档定义
  - 假设项比例合理
  - 支持导出 pytest/postman（如有需要）

## 样本 5：OpenAPI（错误响应缺失）

- 目标：验证“严禁伪造断言”
- 检查项：
  - 不生成臆测错误码
  - 缺失项统一标记待确认
  - 断言来源显示文档缺失说明
  - 结果中保留假设标记
  - 汇总里提示风险

## 样本 6：Postman Collection（变量较多）

- 目标：验证变量与请求结构解析
- 检查项：
  - URL/Method/Body 提取稳定
  - 变量不会导致关键字段丢失
  - 输出含核心 API 用例字段
  - 断言策略符合文档信息完整度
  - 导出格式符合预期

## 样本 7：规则冲突需求文档

- 目标：验证冲突优先级
- 检查项：
  - 冲突规则被识别
  - 处理逻辑遵循“否定 > 肯定 > 模板”
  - 不输出互相打架的主结论
  - 输出中提示待确认项
  - 有对应反向校验用例
